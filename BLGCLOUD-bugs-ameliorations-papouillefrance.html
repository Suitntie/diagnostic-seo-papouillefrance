Alerte GA4 : 81 % du revenu attribué au canal "Unassigned" → attribution e-commerce cassée

Sur 1 507 € de revenu GA4, 1 225 € (81 %) sont attribués au canal "Unassigned" (source/medium non identifié). Ce canal ne représente que 4 % des sessions (180/4 109) mais concentre 58 % des conversions (30/51). Cela indique un problème d'attribution côté DataLayer : les événements purchase sont probablement envoyés sans session utilisateur liée (ex : callback serveur, page de confirmation rechargée, ou dataLayer poussé avant le tracking GA4). L'Organic Search, qui représente 23 % du trafic avec le meilleur engagement (1min55s, 71,6 %), ne génère que 281 € (18 % du CA). La correction du DataLayer e-commerce (bugs #1-#5) est indispensable pour fiabiliser l'attribution des revenus.

Alerte GA4 : page /cart — 1 410 sessions impactées par le bug removeFromCart

La page panier (/cart) est la 3ème page la plus visitée du site (1 410 sessions/28j). Or le bug DataLayer #1 (removeFromCart avec valeurs nulles) et l'absence d'événement view_cart (bug #6) impactent directement l'analyse du funnel d'achat sur cette page. Avec 785 sessions sur /checkout, le taux d'abandon panier réel est impossible à mesurer sans ces corrections.

Urgence : canonicals cassés → désindexation massive confirmée par GSC

Toutes les fiches produit déclarent un canonical qui retourne une 404. La Search Console confirme : 3 656 pages 404 (URLs courtes des canonicals), validation tentée le 09/02/2026 et échouée le 10/02/2026. En parallèle, 1 334 pages ont un canonical ignoré par Google (il a choisi une autre URL). C'est la cause directe de la chute de trafic (-42 %). La correction de ce bug doit être la priorité absolue n°1.

Ce rapport est destiné à l'équipe BLGCLOUD en charge du développement du site et de l'ERP. Il couvre uniquement les bugs liés au développement (SEO technique + DataLayer e-commerce). Les sujets de configuration (GTM, GA4, Merchant Center) sont gérés en interne par l'équipe Papouille.

Cannibalisation & URLs

# Priorité Composant Bug / Problème Comportement actuel Comportement attendu
U1 CRITIQUE Canonical Toutes les URLs canonical des fiches produit pointent vers des pages 404 Chaque fiche produit est accessible via une URL longue avec chemin catégorie (ex : /articles/repas-21/couverts-375/cuillere-5154) mais déclare un canonical court sans catégorie (ex : /articles/cuillere-5154). Or cette URL courte retourne une 404. Google Search Console confirme : 3 656 pages 404 correspondant à ces URLs courtes (validation échouée le 10/02/2026). Pattern systémique sur l'ensemble du catalogue. Deux options : (a) faire fonctionner les URLs courtes en canonical (redirection ou routage) ou (b) mettre l'URL longue (celle qui marche) comme canonical. Option (b) recommandée car plus simple.
U2 CRITIQUE Sitemap Contradiction sitemap vs canonical Le sitemap envoie à Google les URLs longues (qui fonctionnent) mais ces pages déclarent un canonical court (qui retourne 404). Google reçoit un signal contradictoire : le sitemap dit "indexe cette URL longue", mais la page dit "ma version canonique est cette URL courte qui n'existe pas". Le sitemap et le canonical doivent pointer vers la même URL. Aligner les deux sur l'URL longue (fonctionnelle).
U3 HAUTE Open Graph og:url ≠ canonical sur les fiches produit Sur les fiches produit, og:url pointe vers l'URL longue (ex : /articles/repas-21/couverts-375/cuillere-5154) tandis que canonical pointe vers l'URL courte (/articles/cuillere-5154). Troisième signal contradictoire envoyé aux moteurs. og:url et canonical doivent pointer vers la même URL (l'URL longue fonctionnelle).
U4 HAUTE Sitemap Pages internes/administratives dans le sitemap Le sitemap articles contient 16 pages "frais de port", 15 pages "bon de réduction", et 2 pages "article épuisé/annulé". Ces URLs internes (lignes comptables, remises, frais de livraison) sont indexables et visibles par Google. Elles redirigent vers /articles mais polluent le budget de crawl. Exclure du sitemap toutes les URLs de type frais de port, bon de réduction, divers, articles épuisés. Ajouter un noindex sur ces pages ou les bloquer dans robots.txt.
U5 HAUTE Meta title Page /articles : titre incohérent avec le contenu La page principale de listing (/articles) a pour titre "Puériculture - Chaise haute bébé et Siège Repas" et pour meta description 'Vente Puériculture - Chaise haute bébé...'. Le H1 est "Nos produits". Le title semble hérité d'une sous-catégorie au lieu d'être propre à la page. Le titre doit refléter le contenu réel de la page (ex : "Tous nos produits — Papouille"). Le title et la meta description doivent être spécifiques à chaque page.
U6 MOYENNE Sitemap Sitemap de 4 433 URLs pour ~3 200 produits Le sitemap articles contient 4 433 URLs uniques (hors images) : 4 312 fiches produit + 109 catégories + 12 pages racine. Les catégories/sous-catégories sont mélangées avec les produits dans les mêmes fichiers sitemap. Certains produits obsolètes ou internes gonflent artificiellement le sitemap. Séparer les sitemaps : un pour les catégories, un pour les produits actifs uniquement. Purger les articles épuisés, bons de réduction, frais de port.
U7 HAUTE Routage / robots.txt Routes backend BLGCLOUD accessibles et indexables (13 274 URLs bloquées) Google Search Console détecte 13 274 pages bloquées par robots.txt. Le robots.txt bloque /sale*, /invoice*, /fleet*, /equipment*, /opportunity*, /configurator*, /extranet*. Ces routes back-office ne devraient pas être visibles publiquement. Si Google les découvre, c'est que des liens internes pointent vers ces pages (menus, templates, ou scripts). Le commentaire # disallow indexing for now dans le robots.txt suggère un blocage temporaire non résolu. Bloquer l'accès à ces routes côté serveur (middleware ou firewall applicatif) pour les utilisateurs non connectés. Le robots.txt ne doit pas être la seule barrière. Identifier et supprimer les liens internes vers ces URLs. Retourner un code 403 (Forbidden) plutôt que de compter sur le robots.txt.
U8 MOYENNE Routage URLs malformées avec paramètres encodés dans le chemin Google Search Console détecte des URLs malformées de type /articles/motricite-19/tapis-408%26title=papouillefrance.com où un paramètre &title= est encodé (%26) et collé dans le chemin au lieu d'être un query parameter. Ces URLs génèrent des pages en double avec canonical mismatch. Identifier la source de ces URLs (backlinks malformés, script interne, ou bug de génération de liens) et mettre en place une redirection 301 vers l'URL propre. Ajouter une règle serveur pour nettoyer les URLs contenant %26 dans le chemin.

S1 CRITIQUE Performance CLS (Cumulative Layout Shift) = 0,32 Le score CLS mesuré par PageSpeed Insights est de 0,32. Le seuil "bon" est < 0,1. Causes : images sans dimensions, chargement tardif d'éléments, décalages de mise en page. Google Search Console confirme : 1 541 URLs desktop "à améliorer" + 28 lentes (vs mobile OK : 1 707 rapides). Le problème de performance est surtout côté desktop. CLS < 0,1. Ajouter width et height sur toutes les balises <img>, réserver l'espace des éléments dynamiques (banners, sliders), éviter les insertions tardives dans le DOM.
S2 CRITIQUE Performance Score Performance mobile : 19/100 PageSpeed Insights affiche un score de 19/100 (données CrUX) et 46/100 (Lighthouse lab). Problèmes : 5 scripts synchrones bloquant le rendu, CSS inutilisé, JavaScript en double, Total Blocking Time de 800 ms. Score > 50 minimum. Différer les scripts non critiques (defer/async), supprimer le JS en double, purger le CSS inutilisé, optimiser les images (lazy loading, formats modernes WebP/AVIF).
S3 HAUTE Viewport user-scalable=no bloque le zoom La meta viewport contient maximum-scale=1.0, user-scalable=no qui empêche les utilisateurs de zoomer. Pénalisé par Google (accessibilité) et impacte le score Lighthouse. Retirer maximum-scale=1.0 et user-scalable=no. Viewport : width=device-width, initial-scale=1.0 uniquement.

Balises HTML & Meta

# Priorité Composant Bug / Problème Comportement actuel Comportement attendu
S7 CRITIQUE HTML Homepage : aucune balise <h1> La page d'accueil n'a aucune balise H1. Les seuls headings sont des H2 ("Promotion", "Destockage"), H3, H4 et H5 utilisés de façon désordonnée. Le contenu principal est en H4/H5. Ajouter un <h1> unique et descriptif sur la homepage (ex : "Papouille — Matériel de puériculture pour crèches et collectivités"). Respecter la hiérarchie H1 > H2 > H3.
S9 HAUTE Meta Meta descriptions avec guillemets mal échappés Les meta descriptions contiennent des guillemets simples d'encadrement et des doubles guillemets simples d'échappement : 'De // moustiquaire...d''éveil sur mesure'. Cela s'affiche tel quel dans les résultats Google. Générer les meta descriptions sans guillemets d'encadrement et avec un échappement HTML correct.
S10 HAUTE Meta Page 404 : robots à index, follow La page d'erreur 404 a la meta <meta name="robots" content="index, follow">. Les pages d'erreur indexables polluent l'index Google et gaspillent le budget de crawl. Mettre <meta name="robots" content="noindex"> sur toutes les pages d'erreur (404, 500, etc.).
S11 MOYENNE Meta og:type = "website" sur les fiches produit Toutes les pages (y compris les fiches produit) ont og:type = "website". Les fiches produit devraient être identifiées comme des produits pour un meilleur affichage sur les réseaux sociaux. Mettre og:type = "product" sur les fiches produit et ajouter les attributs OG produit (product:price:amount, product:price:currency).
S12 MOYENNE HTML IDs dupliqués dans le DOM Plusieurs éléments partagent le même attribut id (ex : id="5153", id="js-article-price-5153" apparaissent 2 fois). Les IDs doivent être uniques par spécification HTML. S'assurer que chaque id est unique dans la page. Utiliser des classes CSS ou des data-* attributs pour les éléments répétés.
S13 BASSE Meta Pas de balises twitter:card Aucune page du site ne contient de balise twitter:card. Le partage sur Twitter/X ne génère pas de carte enrichie. Ajouter <meta name="twitter:card" content="summary_large_image"> + twitter:title, twitter:description, twitter:image.
S14 HAUTE Images Images internes cassées (404) Semrush Site Audit détecte 800 images internes cassées : ces images sont référencées dans le HTML (balises <img>, CSS, etc.) mais retournent un 404. Google et les utilisateurs ne peuvent pas voir ces éléments visuels. Nettoyer les références vers les images supprimées ou missing. Mettre en place une image par défaut (placeholder) pour les images manquantes. Vérifier que le chemin d'accès aux images (CDN, répertoire static) est correct dans BLGCLOUD.
S15 HAUTE Serveur Erreurs serveur 5xx Semrush Site Audit détecte 29 pages retournant des erreurs serveur 5xx (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable). Ces pages ne sont pas accessibles et générent une mauvaise expérience utilisateur. Investiguer les logs serveur pour identifier la cause des erreurs 5xx (timeout, exception non gérée, crash de processus, manque de ressources). Corriger les bugs sous-jacents côté BLGCLOUD et ajouter du monitoring pour détecter ces erreurs en production.
S16 HAUTE HTML / Liens Attribut nofollow sur tous les liens internes Semrush Site Audit détecte 3 577 pages avec des liens internes portant l'attribut rel="nofollow". Cela empêche le transfert de PageRank entre les pages du site et bloque Googlebot pour le crawl de ces liens. C'est probablement un attribut ajouté globalement par le template BLGCLOUD. Retirer l'attribut rel="nofollow" de tous les liens internes. Les liens internes doivent être "dofollow" par défaut pour permettre le crawl et la transmission de PageRank. Vérifier le template de génération des liens dans BLGCLOUD.
S17 HAUTE Protocole Liens internes en HTTP au lieu de HTTPS Semrush Site Audit détecte 3 577 pages contenant des liens pointant vers des pages HTTP du même site au lieu de HTTPS. Google considère HTTP et HTTPS comme deux URLs distinctes. Cela provoque des redirections inutiles (301 de HTTP vers HTTPS) et dilue le PageRank. Tous les liens internes doivent utiliser le protocole HTTPS. Vérifier les templates de génération d'URL dans BLGCLOUD et s'assurer que la configuration du base_url ou du app_url est définie en HTTPS. Utiliser des URLs relatives ou des fonctions de génération d'URL qui respectent le protocole courant.

La correction des canonicals, du sitemap et des liens internes doit être traitée en urgence absolue.

Contexte DataLayer (e-commerce)

Les bugs ci-dessous ont été identifiés lors de l'audit du conteneur GTM (GTM-PNR2PTM) et de la migration e-commerce GA4 du site. Ils concernent le code dataLayer côté serveur (BLGCLOUD).

Certains problèmes ont été contournés côté GTM (variables Custom JavaScript) mais nécessitent une correction côté serveur pour une solution pérenne.

Éléments déjà contournés côté GTM

Les points suivants fonctionnent grâce à des variables GTM mais restent à corriger côté BLGCLOUD pour une meilleure fiabilité :

  • Format UA → GA4 : Variable cjs_ua_to_ga4_items convertit le format Universal Analytics en GA4 items.
  • Frais de port : Filtrés automatiquement (exclusion des produits avec id: null).
  • currencyCode manquant : Constante const_Currency = "EUR" utilisée en remplacement.
  • Calcul du value : Variable cjs_ecom_value calcule le total à partir des produits.
  • remove_from_cart : Tag GA4 créé avec les mêmes variables de conversion (Version 712).

Bugs DataLayer à corriger

# Priorité Composant Bug / Problème Comportement actuel Comportement attendu
1 CRITIQUE DataLayer removeFromCart — données nulles sur /cart Quand on supprime un produit depuis la page panier, l'événement removeFromCart pousse ecommerce.remove.products avec toutes les valeurs à null (id, name, brand, category, price, quantity). L'objet products[] doit contenir les vraies données du produit supprimé (id, name, brand, category, price, quantity).
2 CRITIQUE DataLayer removeFromCart — price: 0 sur fiches produit Quand on retire un produit depuis sa fiche article, l'événement removeFromCart envoie price: "0" alors que les autres champs (id, name, brand, category) sont corrects. Le champ price doit contenir le prix réel du produit (Number, HT ou TTC selon la convention du site).
3 HAUTE DataLayer "Frais de port" injecté comme produit Dans les événements checkout et purchase, les frais de port sont ajoutés comme un élément du tableau products[] avec id: null et name: "Frais de port". Les frais de port doivent être envoyés dans le champ dédié actionField.shipping et ne pas apparaître dans le tableau products[].
4 HAUTE DataLayer currencyCode absent de la majorité des événements Le champ ecommerce.currencyCode n'est présent que dans l'événement addToCart. Il est absent de productImpressions, productDetailsViews, removeFromCart, checkout et purchase. Chaque événement e-commerce doit inclure ecommerce.currencyCode: "EUR" conformément à la spec Enhanced Ecommerce.
5 MOYENNE DataLayer Format dataLayer Universal Analytics obsolète Tous les événements e-commerce utilisent le format Enhanced Ecommerce UA (ecommerce.detail.products, ecommerce.add.products, etc.) qui est obsolète depuis juillet 2024. Migrer vers le format natif GA4 (ecommerce.items[] avec item_id, item_name, etc.) pour supprimer la dépendance aux variables de conversion GTM.
6 MOYENNE DataLayer Aucun événement view_cart sur /cart La page panier (/cart) ne pousse aucun événement view_cart dans le dataLayer. Le seul événement présent est removeFromCart (avec des données nulles — cf. bug #1). Un événement view_cart doit être poussé au chargement de la page panier avec la liste des produits présents dans le panier, leur quantité et le total.
7 BASSE DataLayer Pas d'événement productClick / select_item Aucun événement n'est déclenché quand l'utilisateur clique sur un produit dans une liste (catégorie, recherche, etc.). Un événement select_item (ou productClick en UA) doit être poussé au clic sur un produit dans une liste, avec l'index et la liste d'origine.
8 BASSE DataLayer Pas d'événement add_payment_info Aucun événement n'est déclenché quand l'utilisateur sélectionne ou valide son moyen de paiement au checkout. Un événement add_payment_info doit être poussé avec le type de paiement (payment_type) et les articles du panier.
9 BASSE DataLayer Pas d'événement refund Aucun mécanisme n'est en place pour pousser un événement refund (total ou partiel) dans le dataLayer. Un événement refund doit être poussé (côté serveur ou via page de confirmation) avec le transaction_id et, en cas de remboursement partiel, les articles concernés.

Impact des bugs critiques (#1 et #2)

Le tag GA4 remove_from_cart (Version 712) est actif mais envoie des données incomplètes ou nulles à cause de ces bugs côté dataLayer. Les rapports GA4 "Comportement d'achat" seront faussés pour l'étape "Suppression du panier". La correction côté BLGCLOUD est indispensable pour que le tracking soit fiable.

Plan d'action — Corrections faisables via API BLGCLOUD (sans dev serveur)

Analyse de l'API BLGCLOUD (app.blgcloud.com/papouille/api) réalisée le 9 avril 2026. Ce plan liste les corrections SEO exécutables directement via l'API REST, sans modification du code serveur BLGCLOUD.

Résumé de l'exploration API

6 712 articles (article/part) · 12 familles · 118 sous-familles · 4 680 redirections (cms/redirect) · 109 pages dynamiques (cms/dynamicPage) · 27 blocs de contenu (cms/contentBlock)

Priorité 1 — Critique (Impact SEO immédiat)

1.1 — Meta descriptions avec guillemets parasites et suffixe générique

Constat : Toutes les meta descriptions sont entourées de guillemets simples ('...') et contiennent un suffixe identique : "Articles de puériculture, modules d'éveil sur mesure, textiles bébé...". Les apostrophes internes sont doublées (''). Résultat : Google ignore ces meta descriptions et génère les siennes.

Cause : Le template BLGCLOUD encapsule la valeur dans des quotes SQL-style et concatène un texte fixe. Ce n'est pas modifiable via API — c'est un bug de template côté BLGCLOUD.

Action BLGCLOUD : Corriger le template de génération des balises <meta name="description"> : supprimer les guillemets englobants, supprimer le doublement des apostrophes, supprimer le suffixe générique. Utiliser uniquement le champ descriptionPubliqueSite (customField) ou comment tronqué à 160 caractères.

Impact : 100 % des pages produit + catégorie + famille affectées.

1.2 — Aucun balisage JSON-LD (structured data) sur les pages produit

Constat : Zéro <script type="application/ld+json"> sur l'ensemble du site. Aucune page (produit, catégorie, homepage) ne comporte de données structurées. Conséquence : aucun rich snippet (prix, disponibilité, avis, breadcrumbs) dans les résultats Google.

Action BLGCLOUD : Injecter automatiquement les schémas JSON-LD suivants :

PageSchéma requisDonnées API disponibles
Page produitProduct + Offerlabel, price (salePrice), brand (make), image (articlePicture), description (descriptionPubliqueSite), reference, category (subfamily)
Page catégorieBreadcrumbList + CollectionPagefamily.title, subfamily.title, URL hierarchy
HomepageOrganization + WebSite + SearchActionCMS settings, contentBlocks
Toutes pagesBreadcrumbListBreadcrumbs déjà rendus en HTML mais sans schéma

Impact : Condition sine qua non pour les rich snippets Google Shopping / Product.

1.3 — Titles de catégorie/famille incorrects (sous-catégorie aléatoire + ordre inversé)

Constat : Deux problèmes dans les balises <title> :

1. Les pages famille incluent une sous-catégorie apparemment aléatoire. Exemple : /articles/mobilier-11 affiche "Mobilier - Meuble en mousse et Pouf | Papouille" au lieu du titre de la famille seule.

2. Le format actuel des pages sous-famille est "Famille - Sous-famille | Papouille" (ex : "Repas - Assiettes Enfants et à Compartiments"). Le format correct demandé par Papouille est Sous-catégorie / Catégorie :

Format attendu : {subfamily.title} / {family.title} | Papouille
Exemple : Assiettes Enfants et à Compartiments / Repas | Papouille
Exemple : Table crèche et scolaire / Mobilier | Papouille

Action BLGCLOUD :

— Pages famille : utiliser uniquement family.title, sans injecter un subfamily.title arbitraire.

— Pages sous-famille : inverser l'ordre pour afficher subfamily.title / family.title | Papouille (et non family.title - subfamily.title).

Impact : 12 pages famille + 118 pages sous-famille.

1.4 — Pages d'erreur indexables (pas de noindex)

Constat : Les pages d'erreur (ex : sous-catégorie supprimée /articles/repas-21/vaisselle-enfant-bebe-342) retournent un titre "Erreur | Papouille" mais avec robots: index, follow. Google indexe ces pages d'erreur.

Action BLGCLOUD : Ajouter <meta name="robots" content="noindex, follow"> sur toutes les pages d'erreur. Idéalement retourner un code HTTP 404 ou 410.

Impact : Pollution de l'index Google, dilution du crawl budget.

Priorité 2 — Haute

2.1 — Canonical URL incohérent avec l'URL accédée

Constat : L'URL /articles/family-X/subfamily-Y/product-NNNNN a un canonical vers /articles/product-slug-NNNNN (sans le chemin catégorie). C'est un choix acceptable MAIS : le système de routage redirige certaines URLs vers le mauvais produit (ex : l'URL "table-rectangulaire" affiche un "matelas à langer" car seul l'ID est utilisé). L'URL affichée est alors trompeuse pour l'utilisateur et Google.

Action BLGCLOUD : Implémenter une redirection 301 vers le canonical correct quand le slug dans l'URL ne correspond pas au produit affiché (basé sur l'ID). Cela empêche les URLs trompeuses d'être indexées.

2.2 — 65 sous-familles sans description SEO (sur 118)

2.3 — 304 articles sans descriptionPubliqueSite + 154 sans comment

Constat : Sur 4 680 redirections, 166 pointent vers / (homepage). Ce sont des produits supprimés redirigés vers l'accueil — considéré comme un soft 404 par Google. Il vaudrait mieux les rediriger vers la catégorie parente.

Priorité 3 — Moyenne

Action BLGCLOUD : Ajouter un <h1> visible ou en sr-only dans le template homepage.

3.2 — Open Graph incomplet sur les pages produit

Constat : Pas de og:price:amount, pas de og:image spécifique au produit. Le og:title affiche "Papouille" au lieu du nom du produit.

Action BLGCLOUD : Compléter les balises Open Graph dans le template produit avec les données article (label, image, prix, catégorie).

3.3 — Pages dynamiques (109) : title = description (cannibalisation)

Constat : Sur les 109 pages dynamiques CMS, de nombreuses ont un title identique à la description. Elles mériteraient des descriptions uniques et optimisées.

Action via API : FAISABLE Modifier via PATCH /cms/dynamicPage/{id} le champ description (max 512 caractères).

3.4 — Hreflang absent (site monolingue fr_FR)

Constat : Aucune balise hreflang. Le site est monolingue mais devrait au moins déclarer hreflang="fr" et un x-default.

Action BLGCLOUD : Ajouter <link rel="alternate" hreflang="fr" href="{canonical}"> dans le template <head>.

3.5 — 2 chaînes de redirection détectées

Constat : 2 URLs redirigent vers une URL qui est elle-même redirigée (redirect chain). Cela ralentit le crawl et perd du link juice.

Plan d'action — Corrections GTM / DataLayer (côté client)

Corrections implémentables via Google Tag Manager (GTM-PNR2PTM) ou le contentBlock trackingCode (cms/contentBlock/6), sans modification du code serveur.

Priorité 1 — Critique

GTM-1 — Attribution e-commerce cassée (81 % revenue "Unassigned")

Constat GA4 : 81 % du revenu est attribué au canal "Unassigned" dans GA4. Le DataLayer envoie les événements e-commerce (productImpressions, productDetailsViews) mais l'attribution de la source/medium est perdue.

Cause probable : Les événements purchase et add_to_cart ne transmettent pas correctement le transaction_id ou le mapping UA → GA4 est incomplet. Le DataLayer utilise encore le format Enhanced Ecommerce (UA) avec ecommerce.impressions au lieu du format GA4 (ecommerce.items).

Action GTM :

1. Migrer les événements DataLayer du format Universal Analytics vers GA4 :
— productImpressions → view_item_list avec ecommerce.items[]
— productDetailsViews → view_item avec ecommerce.items[]
— addToCart → add_to_cart
— removeFromCart → remove_from_cart
— purchase → purchase avec transaction_id, value, currency: "EUR"

2. Vérifier que dataLayer.push({ecommerce: null}) est appelé avant chaque nouvel événement e-commerce.

3. Configurer le tag GA4 avec les paramètres send_page_view et les déclencheurs e-commerce correctement mappés.

Impact : Restaure l'attribution des 1 507 € de revenu mensuel et permet l'analyse des canaux d'acquisition.

GTM-2 — Consentement cookies incomplet (CookieScript)

Constat : Le DataLayer montre des événements CookieScriptConsentUpdated[functionality,strict] mais aucun consentement analytics ou marketing. Les catégories de cookies sont limitées à "functionality" et "strict".

Action GTM : Configurer le Consent Mode v2 dans GTM avec les catégories Google (analytics_storage, ad_storage, ad_user_data, ad_personalization) mappées sur les catégories CookieScript. Sans cela, GA4 perd des données et la conformité RGPD est incomplète.

Priorité 2 — Haute

GTM-3 — Événement purchase : prix à 0 € dans le DataLayer

Constat : L'événement productDetailsViews montre "price": 0 pour certains produits (ex : produits sur devis). Google Analytics enregistre des conversions à 0 € qui faussent le revenu.

Action GTM : Filtrer les événements purchase avec value = 0 ou enrichir le prix via un lookup table GTM basé sur l'ID produit.

GTM-4 — Pas de suivi des recherches internes

Action GTM : Créer un tag GA4 Event search déclenché sur le paramètre URL de recherche (ex : ?q=) avec le terme de recherche en paramètre search_term.

GTM-5 — Tracking code GTM dans un contentBlock modifiable

Constat : Le noscript GTM est dans le contentBlock trackingCode (cms/contentBlock/6). Le script GTM principal est probablement dans le template <head>. Vérifier que les deux parties sont synchronisées (même container ID GTM-PNR2PTM).

Action : Via API, vérifier et mettre à jour si nécessaire le contenu du contentBlock 6 via PATCH /cms/contentBlock/6. FAISABLE
