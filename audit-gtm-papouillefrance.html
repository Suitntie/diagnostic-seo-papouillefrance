7 corrections ont été appliquées : 3 dans GTM (publiées en Version 709) et 4 dans le dashboard CookieScript (Auto-Detect activé, options RGPD corrigées, renouvellement des cookies forcé, 2 cookies Poptin classifiés en Functionality). Le nouveau scan (07/04/2026) a détecté 44 cookies — 0 cookie non classifié.

Audit approfondi GA4 : la configuration de la propriété est globalement solide (Google Signals, attribution data-driven, associations Google Ads + Search Console + Merchant Center, consentement 4/4). Trois optimisations identifiées : activer le scroll tracking, élargir la détection des données utilisateurs (Enhanced Conversions), et vérifier la dimension personnalisée "Devis".

Corrections appliquées et publiées (Versions 709, 710 & 711)

1. GAds - Conversion Tracking — Consent manquant

Le tag de suivi des conversions Google Ads (tag #717) n'avait aucun consentement configuré dans "Additional Consent Checks". Il pouvait théoriquement se déclencher sans le consentement de l'utilisateur, ce qui est contraire au RGPD.

Le tag CMP (tag #702) n'avait pas l'option url_passthrough activée. Sans cette option, les paramètres de clic Google Ads (gclid, dclid) sont perdus lorsque ad_storage est refusé, réduisant fortement l'attribution des conversions.

Le pixel Facebook (tag #14) utilisait analytics_storage comme consentement requis. Or, c'est un pixel marketing/publicitaire qui devrait exiger ad_storage.

analytics_storage ad_storage

Cause racine — Perte de données GA4

⚠ Problème critique : Anciens cookies CookieScript incomplets

L'analyse du site en production a révélé que certains visiteurs possèdent un ancien cookie CookieScriptConsent dont le champ categories ne contient que ["functionality"], sans les catégories "performance" (nécessaire pour analytics_storage) ni "targeting" (nécessaire pour ad_storage).

Conséquence : Pour ces utilisateurs, même après avoir cliqué "Accepter tout", le Google Consent Mode maintient analytics_storage = denied, ce qui bloque totalement GA4.

Flux de consentement (Google Consent Mode v2)

Voici la chaîne de dépendances qui doit fonctionner pour que GA4 collecte des données :

1. Page load → GTM charge CookieScript CMP (Consent Initialization)
2. CMP envoie Default Consent → analytics_storage = "denied"
3. Utilisateur clique "Accepter tout"
4. CookieScript écrit cookie avec categories: ["targeting","performance","functionality"]
5. CookieScript envoie Consent Update → analytics_storage = "granted"
6. GA4 Configuration détecte analytics_storage = "granted" → commence la collecte
✗ Si étape 4 échoue (ancien cookie incomplet) → étape 5 n'envoie pas le bon update → GA4 reste bloqué

🛠️ Implémentation GTM — Approche "Transformation Layer"

L'objectif est de ne rien modifier sur le site. On crée dans GTM des variables et tags qui transforment le format UA en format GA4 à la volée.

Étape 1 — Variables Custom JavaScript (dataLayer → items[])

Créer 4 variables Custom JavaScript dans GTM qui transforment les tableaux UA en tableaux GA4 items[] :

  • cjs_impressions_to_items : lit ecommerce.impressions → retourne items[] avec item_id, item_name, item_brand, item_category, price, index
  • cjs_products_to_items : lit ecommerce.detail.products ou ecommerce.add.products → retourne items[]
  • cjs_checkout_to_items : lit ecommerce.checkout.products → retourne items[], en filtrant les produits avec id === null (ex: "Frais de port")
  • cjs_ecom_value : calcule value = somme de (price × quantity) pour chaque produit (hors frais de port)

Seul événement absent : remove_from_cart (optionnel, impact mineur).

⚠️ Point d'attention — Filtrage "Frais de port"

Le site inclut "Frais de port" comme un produit dans les tableaux checkout.products[] et purchase.products[] (avec id: null, category: "Frais de port"). La variable Custom JavaScript cjs_checkout_to_items doit filtrer cet item pour ne pas fausser les rapports produits GA4. Filtre : products.filter(p => p.id !== null).
